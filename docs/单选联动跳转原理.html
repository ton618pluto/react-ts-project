场景设定

假设问卷中有 4 个组件：问题A(questionRadio) → 问题B(questionTitle) → 问题C(questionRadio) → 问题D(questionInput)。其中问题A的选项1配置为跳转到问题D。

1

第1步：点击画布中的问题A，选中组件

用户操作：点击问题A组件
事件：onClick → dispatch(changeSelectedId('问题A的fe_id'))
状态变化：Redux: selectedId = '问题A的fe_id'
效果：问题A显示选中边框（selected样式），右侧属性面板显示问题A的RadioPropComponent

selectedValues 是什么？
selectedValues: Record<string, string> = { [componentFeId]: selectedOptionValue }
- key = 组件的 fe_id（每个 questionRadio 的唯一标识）
- value = 用户选择的选项的 value（选项的业务值，不是跳转目标）

示例：用户选了"问题A"的"选项1"，然后选了"问题B"的"选项X"
selectedValues = { '问题A的fe_id': '选项1的value', '问题B的fe_id': '选项X的value' }

作用：
1. 控制 Radio.Group 高亮显示哪个选项（传给 Radio 的 value prop）
2. 在跳转算法中查找 jumpTo[selectedValue] 得到跳转目标 fe_id
3. 临时状态，退出预览模式即销毁，不持久化到后端
2

第2步：在右侧属性面板中，为问题A的选项1选择跳转目标

用户操作：点击选项1对应的Select下拉框，选择"问题D"
事件链： 

Select onChange('问题D的fe_id')
  → form.setFieldValue('options', currentOptions)
     (currentOptions[0].jumpTo = '问题D的fe_id')
  → handleValuesChange()
     → form.getFieldsValue() 获取完整表单数据
     → 遍历 rawOptions，提取每个 option 的 jumpTo
     → 构建 newJumpTo = { '选项1的value': '问题D的fe_id' }
     → fieldsValue.jumpTo = newJumpTo
  → onChange(fieldsValue)
     → handlePropsChange(fieldsValue)
        → dispatch(changeComponentProp({ fe_id: '问题A的fe_id', newProps: fieldsValue }))
           → Redux componentsList 中找到问题A
           → comp.props.jumpTo = { '选项1的value': '问题D的fe_id' }
关键：newJumpTo = { ...jumpTo }
这里用 { ...jumpTo } 先复制已有的 jumpTo，是为了保留其他选项的跳转配置。如果直接 newJumpTo = {} 会覆盖掉之前配置的其他选项跳转。
3

步骤3：点击工具栏的"预览"按钮，进入预览模式

用户操作：点击工具栏的预览按钮
事件：dispatch(togglePreviewMode())
状态变化：Redux: previewMode = true
效果：EditCanvas 检测到 previewMode=true，渲染 PreviewCanvas 而不是 SortableContainer 

// EditCanvas 中的条件渲染

预览模式下组件可以响应鼠标事件（点击选项），编辑模式下不行。
4

步骤4：PreviewCanvas 渲染，执行跳题算法

初始状态：selectedValues = {}（空对象）
selectedValues 含义：用户的选项选择记录，{ [fe_id]: 选择的选项value }
visibleComponents useMemo 计算： 

componentsList = [问题A, 问题B, 问题C, 问题D]
selectedValues = {}  // 暂无选择

// 遍历：
i=0: 问题A → type='questionRadio' → jumpTo['选项1']='问题D的fe_id'
       selectedValues['问题A的fe_id'] = undefined → 不满足跳转条件
       → result.push(问题A)
i=1: 问题B → → result.push(问题B)
i=2: 问题C → → result.push(问题C)
i=3: 问题D → → result.push(问题D)

// 结果：visibleComponents = [问题A, 问题B, 问题C, 问题D]
// 因为 selectedValues 为空，没有触发跳转，所有题目正常显示
注意：刚进入预览模式时 selectedValues 是空的，所以所有题目都会显示。用户必须先点击某个单选题选项，才会触发跳转逻辑。
5

步骤5：用户点击问题A的"选项1"

用户操作：点击问题A的 Radio 选项1
事件链： 

1. Radio.Group onChange(e)
   → e.target.value = '选项1的value'

2. QutstionRadio 组件接收
   → onChange({ ...props, value: '选项1的value' })
   // props 包含 jumpTo: { '选项1的value': '问题D的fe_id' }

3. PreviewCanvas 中注入的 onChange
   → (newProps) => {
        if (newProps.value) {
          handleOptionChange('问题A的fe_id', newProps.value)
        }
      }

4. handleOptionChange('问题A的fe_id', '选项1的value')
   → setSelectedValues(prev => ({
       ...prev,
       '问题A的fe_id': '选项1的value'
     }))
   // selectedValues 从 {} 变成 { '问题A的fe_id': '选项1的value' }

5. selectedValues 变化 → useMemo 依赖变化 → 重新计算 visibleComponents
6

步骤6：useMemo 重新计算 visibleComponents，触发跳转

新状态：selectedValues = { '问题A的fe_id': '选项1的value' }
重新计算： 

componentsList = [问题A, 问题B, 问题C, 问题D]
selectedValues = { '问题A的fe_id': '选项1的value' }

for (let i = 0; i < componentsList.length; i++) {
  i=0: 问题A → type='questionRadio'
       jumpTo = { '选项1的value': '问题D的fe_id' }
       selectedValue = selectedValues['问题A的fe_id'] = '选项1的value'
       // 有 jumpTo 且有 selectedValue
       → targetFeId = '问题D的fe_id'
       → targetIndex = 3 (问题D在数组中的index)
       → targetIndex(3) > i(0)，满足跳转条件
       → result.push(问题A)
       → i = targetIndex - 1 = 2
       → continue (跳过 i++，下一轮 i=3)

  i=3: 问题D → type='questionInput'
       → result.push(问题D)

  // 问题B和问题C被跳过了！
}

visibleComponents = [问题A, 问题D]
// 渲染结果：只显示问题A和问题D，问题B和问题C被隐藏
i = targetIndex - 1 的补偿法：
因为 for 循环在每次迭代结束后会自动执行 i++。设置 i = targetIndex - 1 后，下一轮循环开始时 i++ 会变成 targetIndex，正好跳到目标位置。如果用 i = targetIndex，则会跳到 targetIndex+1，漏掉目标。
7

步骤7：React 检测到 visibleComponents 变化，重新渲染

React 行为：visibleComponents 从 [问题A, 问题B, 问题C, 问题D] 变成 [问题A, 问题D]
渲染：只渲染问题A和问题D的问题卡片
Radio 显示：问题A的 Radio.Group 会高亮显示"选项1"（因为传入了 value prop） 

// PreviewCanvas 中为 questionRadio 注入 value 和 onChange
if (type === 'questionRadio') {
  const curValue = selectedValues[fe_id] || mergedProps.value
  // curValue = '选项1的value'（从 selectedValues 读取的）

  return (
    <Component
      {...mergedProps}
      value={curValue}  // 控制 Radio 高亮显示哪个选项
      onChange={(newProps) => {
        if (newProps.value) {
          handleOptionChange(fe_id, newProps.value)
        }
      }}
    />
  )
}

场景2：多个单选题联动

假设问卷组件顺序：[问题A(跳D), 问题B, 问题C(跳F), 问题D, 问题E, 问题F]

问题A的选项1 → 跳D；问题C的选项2 → 跳F

用户操作：先选问题A的选项1，再选问题C的选项2

selectedValues = {
  '问题A的fe_id': '选项1的value',
  '问题C的fe_id': '选项2的value'
}

// 遍历：
i=0: 问题A → jumpTo['选项1']='问题D的fe_id' → targetIndex=3 → i=3-1=2 → continue
i=3: 问题D → → result.push(问题D)
i=4: 问题E → → result.push(问题E)
i=5: 问题F → jumpTo['选项2']='问题F自己的fe_id'(假设) 或 另一个目标
       如果选项2跳转到问题E → targetIndex=4 → i=4-1=3 → continue

visibleComponents = [问题A, 问题D, 问题E, 问题F] 或根据跳转配置变化

// 注意：问题B和问题C被跳过了（问题B在问题A跳D的路上，问题C在问题C自己跳F的路上）
多选题关键点：
1. selectedValues 用组件 fe_id 作为 key，所以不同单选题的选择互不影响
2. 每道题的 jumpTo 存在各自的 props.jumpTo 里，各自独立
3. 遍历是逐个处理的，每道题根据自己的 selectedValues 和 jumpTo 决定跳到哪里
4. 如果两道单选题的跳转目标形成链式（如A跳B，B跳C），算法会继续处理
5. 只能向后跳，不允许向前跳（防止循环）

WHAT
🎯

功能是什么？

在问卷编辑器中，为单选题的每个选项配置"跳转到哪个题目"。预览模式下，用户选择选项后，中间题目被跳过，直接跳到目标题目。

HOW
⚡

怎么实现的？

Redux 存储 jumpTo 配置 → Form.List 编辑配置 → PreviewCanvas 用 useMemo 计算 visibleComponents → 拦截 onChange 更新 selectedValues → 触发重新计算。

📦 REDUX STORE

Redux状态管理：如何存储jumpTo配置

📦
组件状态结构

Redux 中每个问卷组件的完整数据结构如下。关键字段是 jumpTo: Record<string, string>，存储"选项值 → 目标fe_id"的映射。

questionTypes.ts 
// 单选题 Props 类型定义
export type QuestionRadioPropsType = {
  title?: string
  isVertical?: boolean
  value?: string              // 当前选中值
  options?: OptionType[]       // 选项列表
  isLocked?: boolean
  onChange?: (props: QuestionRadioPropsType) => void
  jumpTo?: Record  // ★ 核心：key=选项value, value=目标组件fe_id
}

// 单个问卷组件的完整数据结构
export type ComponentInfoType = {
  fe_id: string                // 前端生成的唯一ID
  title: string
  type: string                 // 'questionRadio' | 'questionInput' ...
  isHidden: boolean
  isLocked: boolean
  props: QuestionRadioPropsType  // ★ 包含 jumpTo
}
componentsReducer/index.ts 
// Redux Store 组件状态
export type ComponentStateType = {
  componentsList: ComponentsListType   // 所有组件
  selectedId: string                    // 当前选中组件ID
  copiedComponent: ComponentInfoType | null
  previewMode: boolean                  // ★ 预览模式开关
}

// changeComponentProp reducer - 更新组件属性
changeComponentProp(state, action: PayloadAction<{ fe_id: string; newProps: ComponentPropsType }>) {
  const { fe_id, newProps } = action.payload
  const comp = state.componentsList.find(c => c.fe_id === fe_id)
  if (comp) {
    comp.props = { ...comp.props, ...newProps }  // ★ newProps 包含 jumpTo
  }
}
为什么 jumpTo 存在 props 里而不是独立管理？
因为 props 是组件的"属性"，jumpTo 本质上是单选题的业务配置，和 title、options 同级。自然放在 props 中，随组件Props一起更新和持久化。
📝 FORM.LIST

属性面板：Form.List动态表单与jumpTo存取

🔧
RadioPropComponent 完整流程

AntD 的 Form.List 是实现动态选项列表的核心。它允许我们在表单中渲染可变数量的表单项，每个选项有自己的输入框和跳转Select。

RadioPropComponent.tsx (关键代码) 
const RadioPropComponent: FC = (props) => {
  const { title, isVertical, value, options = [], onChange, isLocked, jumpTo = {} } = props
  const [form] = Form.useForm()
  const { visibleComponnents, selectedId } = useGetComponentInfo()  // 获取所有组件用于可选跳转目标

  // 构建可选跳转目标下拉列表
  const jumpTargetOptions = visibleComponnents
    .filter(comp => comp.fe_id !== selectedId)  // 排除自己
    .map(comp => ({
      value: comp.fe_id,
      label: comp.title || comp.props?.text,
    }))

  // ★ 关键：handleValuesChange 构建新的 jumpTo 对象
  function handleValuesChange() {
    if (onChange) {
      const fieldsValue = form.getFieldsValue()
      const rawOptions = fieldsValue.options || []

      const newOptions: OptionType[] = []
      const newJumpTo: Record = { ...jumpTo }  // ★ 先复制已有jumpTo

      rawOptions.forEach((opt) => {
        if (!opt || opt.text === undefined) return

        const option = opt as OptionType
        if (!option.value) option.value = nanoid()
        newOptions.push({ value: option.value, text: option.text })

        // ★ 从 option 对象直接读取 jumpTo（Select onChange 时写入的）
        const jumpToVal = (opt as Record).jumpTo
        if (jumpToVal && typeof jumpToVal === 'string') {
          newJumpTo[option.value] = jumpToVal
        }
      })

      fieldsValue.options = newOptions
      fieldsValue.jumpTo = newJumpTo  // ★ 挂到 fieldsValue 上

      onChange(fieldsValue)  // ★ 触发 Redux 更新
    }
  }

  return (
    <Form onValuesChange={handleValuesChange}>
      <Form.List name="options">
        {(fields, { add, remove }) => {
          return fields.map(({ key, name }) => (
            <Space key={key} align="baseline">
              <Form.Item name={[name, 'text']}>
                <Input />
              </Form.Item>
              {/* ★ 跳转目标 Select */>
              <Select
                options={jumpTargetOptions}
                onChange={val => {
                  const currentOptions = form.getFieldValue('options') || []
                  currentOptions[name] = {
                    ...currentOptions[name],
                    jumpTo: val || undefined,  // ★ 写入当前选项的jumpTo
                  }
                  form.setFieldValue('options', currentOptions)
                  handleValuesChange()  // ★ 触发保存到 Redux
                }}
              />
            </Space>
          ))
        }}
      </Form.List>
    </Form>
  )
}
为什么必须先复制 { ...jumpTo } 再修改？
因为 handleValuesChange 在每次任何选项变化时都触发。如果直接构建 newJumpTo = {}，会导致每次都丢失之前配置的跳转目标。用 ...jumpTo 先展开已有配置，再覆盖本次修改的选项，才能保留所有选项的跳转配置。
⚡ ALGORITHM

预览模式：visibleComponents 跳题算法

⚡
核心：useMemo + 手动索引循环

不能用普通的 map/filter，因为跳题后需要"跳过"某些元素，所以用 for 循环 + 手动 i++。通过 i = targetIndex - 1 实现跳到目标位置。

EditCanvas.tsx - PreviewCanvas 
function PreviewCanvas({ componentsList }) {
  const [selectedValues, setSelectedValues] = useState>({})

  // ★ useMemo 计算可见组件列表
  const visibleComponents = useMemo(() => {
    const result: ComponentInfoType[] = []

    for (let i = 0; i < componentsList.length; i++) {
      const comp = componentsList[i]
      const compConf = getComponentConfByType(comp.type)
      if (!compConf) continue

      const mergedProps = { ...compConf.defaultProps, ...comp.props }

      // ★ 关键：如果是单选题且有选择值，查 jumpTo
      if (comp.type === 'questionRadio') {
        const jumpTo = mergedProps.jumpTo
        const selectedValue = selectedValues[comp.fe_id]

        if (jumpTo && selectedValue && jumpTo[selectedValue]) {
          const targetFeId = jumpTo[selectedValue]
          const targetIndex = componentsList.findIndex(c => c.fe_id === targetFeId)

          if (targetIndex > i) {  // ★ 只允许向后跳
            result.push(comp)
            i = targetIndex - 1   // ★ 关键：跳到目标位置，循环末尾会 i++
            continue
          }
        }
      }

      result.push(comp)  // 普通组件或无需跳转的组件直接入栈
    }

    return result
  }, [componentsList, selectedValues])

  function handleOptionChange(componentFeId: string, value: string) {
    setSelectedValues(prev => ({ ...prev, [componentFeId]: value }))
  }

  // 渲染时为单选题注入自定义 onChange
  return (
    <div className={styles.canvas}>
      {visibleComponents.map(item => {
        if (item.type === 'questionRadio') {
          const curValue = selectedValues[item.fe_id] || item.props.value
          return (
            <Component
              {...item.props}
              value={curValue}  // ★ 控制 Radio.Group 显示选中状态
              onChange={(newProps) => handleOptionChange(item.fe_id, newProps.value)}
            />
          )
        }
        return <Component {...item.props} />
      })}
    </div>
  )
}

跳题算法图解：假设 [A,B,C,D]，B的选项1跳转到D

i=0, 处理A → result=[A]
i=1, 处理B → jumpTo['选项1']='D的fe_id' → targetIndex=3 → i=3-1=2 → continue
i=3, 处理D → result=[A,B,D]
C 被跳过了！✅

为什么用 for 而不用 while？
while 更直接，但 for 在 JS 中更常见且易于理解。关键是 i = targetIndex - 1 后循环末尾 i++ 会变成 targetIndex，正好跳到目标位置。如果用 i = targetIndex，则循环末尾 i++ 会跳到 targetIndex+1，漏掉目标题目。
🔄 ONCHANGE CHAIN

React组件间onChange事件传播链路

┌─────────────────────────────────────────────────────────┐ │ 用户点击 Radio 选项 │ └────────────────────────┬────────────────────────────────┘ ↓ ┌─────────────────────────────────────────────────────────┐ │ Radio.Group (AntD) 触发 onChange(e) │ │ e.target.value = 'item1' (选项的value) │ └────────────────────────┬────────────────────────────────┘ ↓ ┌─────────────────────────────────────────────────────────┐ │ QutstionRadio 组件接收 onChange prop │ │ onChange({ ...props, value: e.target.value }) │ └────────────────────────┬────────────────────────────────┘ ↓ ┌─────────────────────────────────────────────────────────┐ │ ★ PreviewCanvas 中注入的 onChange: │ │ (newProps) => { │ │ if (newProps.value) { │ │ handleOptionChange(fe_id, newProps.value) │ │ } │ │ } │ └────────────────────────┬────────────────────────────────┘ ↓ ┌─────────────────────────────────────────────────────────┐ │ setSelectedValues(prev => ({ ...prev, [fe_id]: value }))│ │ ★ 更新临时状态，触发 useMemo 重新计算 │ └─────────────────────────────────────────────────────────┘
层级组件onChange职责
1Radio.Group (AntD)原生 onChange，传递 e.target.value
2QutstionRadio接收 onChange prop，调用 onChange({...props, value})
3PreviewCanvas注入 onChange → handleOptionChange → setSelectedValues
4selectedValues state触发 useMemo 重新计算 visibleComponents
为什么 Radio.Group 需要 value prop？
AntD 的 Radio.Group 是"受控组件"，需要 value prop 来决定显示哪个选项被选中。如果不传 value，即使调用了 onChange，Radio.Group 也不知道当前应该高亮哪个选项，UI 不会更新。
🖱️ CSS POINTER-EVENTS

CSS pointer-events：编辑/预览模式的交互切换

🎨
编辑模式 vs 预览模式

编辑模式下，画布上的组件只应响应"选中"操作（点击选中组件），不应响应组件自身的交互（如点击单选题选项）。预览模式则相反，需要组件完全可交互。

EditCanvas.module.scss 
.canvas {
  overflow: auto;

  .component-wrapper {
    margin: 12px;
    padding: 12px;
    border: 1px solid var(--color-border);
    border-radius: var(--border-radius);

    &:hover {
      border-color: var(--color-primary);
      box-shadow: var(--shadow-sm);
    }
  }

  // ★ 编辑模式：禁用组件内部交互，鼠标事件穿透到 wrapper
  .component {
    pointer-events: none;  // 组件不接收鼠标事件
  }

  // ★ 预览模式：恢复组件交互
  &.preview-mode .component-wrapper {
    pointer-events: auto;  // wrapper 可以点击（选中组件）
  }

  &.preview-mode .component {
    pointer-events: auto;  // 组件本身也可以点击（选项选择）
    position: relative;
    z-index: 1;            // 建立层叠上下文，确保事件不被父级捕获
  }
}

.selected {
  border-color: var(--color-primary) !important;
  box-shadow: var(--shadow-sm);
}

.locked {
  opacity: 0.5;
}
为什么 pointer-events: none 加在 .component 而不是 .component-wrapper？
因为编辑模式下，我们需要：1) wrapper 能响应点击（选中组件）2) 组件内部不响应点击（避免触发选项选择）。如果给 wrapper 设 none，wrapper 本身就不能点击了，选中功能失效。所以 none 要加在 .component 上，让事件"穿透"组件，抵达 wrapper。

✅ 优点

  • 纯 CSS 方案，无需 JS 判断，性能好
  • 编辑/预览模式切换只需切换一个 class
  • 不破坏组件树结构

⚠️ 注意点

  • 子元素设置了 pointer-events 会覆盖父级
  • position: relative/z-index 用来建立新的堆叠上下文
  • AntD 组件内部可能自带 pointer-events 样式
🎯 KEY POINTS

关键技术难点与解决方案

难点1
🔢

Form.List 中动态字段的 jumpTo 存储

问题：Form.List 的每个选项是动态字段，Select 的值不是表单标准字段结构。

解决：在 Select onChange 时，将 jumpTo 写入对应 option 对象的 custom 属性，通过 form.setFieldValue('options', currentOptions) 整体更新，然后在 handleValuesChange 中从 rawOptions 里读取每个 option 的 jumpTo 拼成新对象。

难点2
💾

每次 handleValuesChange 覆盖之前配置的 jumpTo

问题：handleValuesChange 用 newJumpTo = {} 从头构建，每次调用只包含当前变更的选项，之前配置的其他选项 jumpTo 会丢失。

解决：const newJumpTo: Record<string, string> = { ...jumpTo }。先展开 props.jumpTo（Redux中的完整数据）作为基础，再遍历 rawOptions 覆盖每个选项的 jumpTo，这样既能保留之前配置，又能合并本次修改。

难点3
📊

跳题后指针 i 的正确跳转

问题：从 index 0 跳到 index 3，直接 i = targetIndex 后循环末尾 i++ 会变成 4，跳过目标。

解决：i = targetIndex - 1，循环末尾执行 i++ 后正好变成 targetIndex，正好处理目标组件。这是典型的"补偿法"。

难点4
🖱️

编辑模式点击选项没反应（pointer-events 陷阱）

问题：.component { pointer-events: none } 阻塞了组件内部所有鼠标事件，包括预览模式下 Radio 的点击。

解决：给 .preview-mode .component 单独设置 pointer-events: auto 覆盖父级规则。同时给 .preview-mode .component-wrapper 也加 pointer-events: auto，确保预览模式下组件本身可以响应交互。

"我实现过一个低代码问卷平台中的'单选题联动跳转'功能。用户可以为每个选项配置'跳转到哪个题目'，预览模式下选择后中间题目会被跳过。

整体方案是：Redux 存储 jumpTo 配置（Record<optionValue, targetFeId>），属性面板用 Form.List 动态渲染选项，每行有个 Select 选择跳转目标，Select onChange 时写入 option 对象的 custom 属性，然后在 handleValuesChange 中从 rawOptions 里读取并合并到 jumpTo 对象，通过 onChange 持久化到 Redux。

预览模式下，PreviewCanvas 用 useMemo 计算可见组件列表。核心算法是：遍历 componentsList，对单选题查找 jumpTo[selectedValue] 得到目标 index，然后 i = targetIndex - 1 + continue 实现跳转。CSS 上通过 .component { pointer-events: none } 实现编辑模式和预览模式的交互切换。

技术难点包括：1）Form.List 动态字段中非标准数据的存储结构；2）每次 handleValuesChange 时如何保留之前配置的 jumpTo；3）跳题后指针的补偿计算。"

❓ Q&A

高频追问

✓ Q: 为什么用 useMemo 而不是 useState + 手动触发重新计算？

A: useMemo 的依赖数组是 [componentsList, selectedValues]。当用户点击选项时，handleOptionChange 更新 selectedValues，自动触发 useMemo 重新计算，无需手动调用。这比 useState + 手动触发更简洁，且 useMemo 有缓存机制，dependencies 没变时直接返回缓存结果，性能更好。

✓ Q: 为什么 jumpTo 的 key 用选项的 value 而不是选项的 index？

A: 因为选项的 value 是业务语义上的唯一标识，而 index 会随着选项的增删改而变化。如果用 index，当用户在选项中间插入一个新选项时，原有的 index 全打乱了，jumpTo 配置就错乱了。用选项 value 作为 key，即使选项顺序变化，只要选项 value 不变，jumpTo 就始终正确。

✓ Q: 跳题算法为什么只允许向后跳，不允许向前跳？

A: 因为向前跳会破坏问卷的线性填写体验。想象一下：用户选了一个选项后跳回前面做过的题目，容易造成循环。更重要的是，向前跳需要"回退"已经显示过的组件，而我们的 visibleComponents 结果数组只进不出，没有回退机制。如果需要支持向前跳，算法需要改用双向链表或者状态机来记录回退路径。

✓ Q: Form.List 的 onChange 和普通 Form.Item 的 onChange 有什么区别？

A: Form.List 的 onValuesChange 会在任何子字段变化时触发，接收的参数是 (changedValues, allValues)。而普通 Form.Item 的 onChange 通常是单个字段变化后立即触发。在 RadioPropComponent 中，我们用 Form 的 onValuesChange 统一处理所有字段变化，手动从 allValues 中提取 options 和 jumpTo，再通过 props.onChange 传给父组件。

✓ Q: 为什么选择用 Select 而不是 InputNumber 来配置跳转？

A: 最初需求是"跳过N个题目"，但后来改成了"跳转到题目X"。两者的区别是：跳过N题是相对跳转，数字会随题目增删变化；跳转到题目X是绝对跳转，目标是固定的 fe_id。Select 下拉框直接展示了所有可选目标，操作直观，不易误输入。用户说"你应该帮我保存啊"，正是因为 Select 选择后需要触发 handleValuesChange 才能持久化到 Redux。
